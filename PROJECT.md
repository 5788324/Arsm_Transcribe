## 1. 项目目标

```
输入: RJ音声文件夹(每个作品一个文件夹,内含 01.mp3, 02.wav, 03.flac ...)
输出: 每个音频对应三个LRC文件
  xxx.ja.lrc          日文原文,带时间戳
  xxx.zh.lrc          中文翻译,带时间戳
  xxx.bilingual.lrc   双语合并,单行格式: [时间戳]日文 / 中文
```

核心链路:**本地音频 → 日文转录(带时间轴) → 中文翻译 → LRC文件**

使用场景:本地音乐播放器(支持LRC的均可)播放RJ音声,边听边看双语歌词。

### 非目标(明确不做的事,防止范围膨胀)
- 不做GUI(第一版纯CLI,够用再说)
- 不做说话人分离
- 不做实时/流式处理(都是离线批处理已有文件)
- 不接云端API做转录或翻译(音频内容隐私限制,全程本地模型)

## 3. 架构设计:为什么转录和翻译要分两步、完全解耦

这是本项目**最重要的架构决策**,后续任何AI改代码都不能违反这一条:

```
音频文件
   │
   ▼ [转录模块] —— 可插拔,支持多个ASR backend
xxx.transcript.json   ← 中间产物,落盘,只生成一次
   │
   ▼ [翻译模块] —— 独立运行,可重复执行
xxx.translated.json   ← 中间产物,落盘
   │
   ▼ [LRC生成模块] —— 纯格式转换,最简单最稳定
xxx.ja.lrc / xxx.zh.lrc / xxx.bilingual.lrc
```

**为什么必须这样设计**:
- 转录是整条链路里最耗时、最依赖GPU的环节。如果转录和翻译写成一条流水线直接出LRC,任何环节出错(比如翻译API超时)都会导致转录的成果作废,必须重跑最贵的那一步。
- 中间产物落盘后,翻译模型可以反复换、反复调prompt重跑,完全不影响转录结果。
- 用户计划后续做"转录模型A/B测试"(faster-whisper vs Qwen3-ASR),如果转录模块本身是可插拔的backend,加一个新模型只是加一个新的adapter,不需要改动翻译和LRC模块。

**所有模块之间只通过磁盘上的JSON文件通信,不允许在内存里直接传递数据跨模块调用。** 这条规则看起来"性能不是最优",但好处是任何一步都能单独调试、单独重跑、单独检查中间结果,对一人维护+多AI接力的项目来说,可调试性比性能更重要。

### 3.1 已知的真实输出格式(实测样本,2026-06-22)

用户已用 cu128-transcribe 整合包(Jim6789日文转录模型 + ASMR专用VAD)对一个真实RJ音频跑出了LRC,以下是从真实输出中确认的格式细节,**asr_runner.py 的解析逻辑必须按这份实测结果设计,不能凭空假设格式**:

```
[00:03.04]い、いらっしゃい。待ってたよ……
[00:06.86]
[00:08.38]どう?この日のために用意した、特別仕様の巫女服。似合ってるでしょ?
```

**关键格式特征**:
- 每句台词后紧跟一个**无文本的时间戳行**,这一行代表"上一句的结束时间(end)",不是新的一句。解析时必须识别这种空文本行并合并到上一句的 `end` 字段,不能当成独立台词处理,否则transcript.json会被这些空行污染。
- 正常情况下相邻两句间隔较短(几秒内),但遇到长停顿/喘息/环境音密集段时,会出现**整段静音被跳过、相邻两句时间差达十几秒甚至更长**的情况(实测样本中出现过17秒的空当)。这种长间隔需要在分段清洗阶段重点检查,可能是VAD漏识别的信号,也可能是真实的长停顿,需要结合实际听感判断。
- 该整合包默认VAD配置在ASMR场景下**疑似存在漏识别风险**:实测样本里多处大段静音区间几乎没有输出文本,需要后续实际听音频核对,如果确认是漏识别,需要调低VAD阈值重跑。
- 句子切分粒度不稳定,有时该连的没连(单个语气词独立成句,如"あ、ゆう……み……っ"),有时该断的没断(长达20-30字的连续呻吟/喘息被拼成一整句)。**这不是ASR模型的问题,是分段(segmentation)逻辑的问题**,不需要靠换模型解决。

### 3.2 分段清洗是独立的必要环节,不是可选项

基于3.1的实测发现,**转录(ASR)和翻译(LLM)之间,必须插入一道"分段清洗"步骤**,处理原始transcript.json,再交给翻译模块。理由:碎片化的单字感叹词翻译模型容易翻得奇怪或跳过,过长的拼接句也不利于生成可读的LRC单行。

```
音频 → [ASR] → 原始transcript.json(可能碎片化/拼接过长)
            → [分段清洗] → 清洗后的transcript.json(合并短句/拆分长句)
            → [翻译] → translated.json
            → [LRC生成]
```

分段清洗规则建议(供Codex实现时参考,具体阈值后续根据实测调整):
- 合并规则:相邻两句间隔小于约0.5秒、且合并后总字数不超过单行上限,则合并为一句
- 拆分规则:单句超过PROJECT.md中`config.yaml`配置的`max_chars_per_line`,在标点处拆分为多句,各自分配合理的时间区间
- 长静音判定:相邻两句间隔超过一定阈值(如10秒以上),标记为"疑似漏识别区间",写入日志供人工核查,不自动臆造内容

## 4. 模块拆分与目录结构

```
RJ-LRC-Local/
├── PROJECT.md              # 本文档
├── WORKLOG.md               # 工作日志(唯一日志文件)
├── config.yaml              # 所有可配置参数集中在这里,不允许硬编码
├── app.py                   # CLI入口
├── input/                   # 待处理的RJ文件夹放这里
├── cache/                   # 中间产物: xxx.transcript.raw.json, xxx.transcript.clean.json, xxx.translated.json
├── output/                  # 最终LRC输出
├── logs/                    # 失败记录、运行日志
└── modules/
    ├── asr/
    │   ├── base.py           # ASR backend的抽象接口(所有backend必须实现这个接口)
    │   ├── faster_whisper_backend.py   # 第一个backend: faster-whisper + ASMR VAD
    │   └── qwen3_asr_backend.py        # 第二个backend(后续A/B测试用,先占位)
    ├── segment_cleaner.py    # 分段清洗:合并短句/拆分长句/标记疑似漏识别区间(见3.2节)
    ├── translate.py          # 翻译模块,调本地OpenAI-compatible API
    ├── lrc_writer.py          # JSON转LRC
    └── batch.py               # 批处理总控,扫描input/下所有文件夹依次处理
```

### 4.1 ASR backend抽象接口(关键约束)

所有ASR backend必须实现统一的输入输出格式,不允许各自为政:

```python
# modules/asr/base.py 的接口约定(伪代码示意,不是最终代码)
def transcribe(audio_path: str, config: dict) -> list[dict]:
    """
    返回格式必须是:
    [
      {"start": 12.34, "end": 15.67, "text": "おはよう", "confidence": 0.92},
      ...
    ]
    start/end 单位是秒(float),不是字符串时间戳
    """
```

第一版只接入 **faster-whisper large-v3 + "海南鸡"(TransWithAI-ChickenRice)的ASMR专用VAD模型**。Qwen3-ASR backend先建空文件占位,等第一版跑通、且用户决定要做A/B对比时再实现——**注意Qwen3-ASR本身不输出时间戳,需要额外接入Qwen3-ForcedAligner-0.6B做对齐,这是后续实现该backend时必须处理的已知约束,不要漏掉。**

### 4.2 config.yaml 示例结构

```yaml
asr:
  backend: faster_whisper        # 切换backend只改这一个字段
  model: large-v3
  device: cuda
  compute_type: float16
  language: ja
  vad:
    model: asmr_optimized        # 海南鸡的ASMR专用VAD,不是标准VAD
    threshold: 0.2                # ASMR耳语场景默认阈值容易误删,需要调低

translate:
  enabled: true
  provider: openai_compatible
  base_url: http://localhost:1234/v1
  model: qwen3-14b
  batch_size: 20                  # 每次API请求送多少句,不要逐句调用

lrc:
  bilingual_format: same_line     # 同一行 "日文 / 中文",比双行更兼容播放器
  max_chars_per_line: 45
```

## 5. 批处理的鲁棒性要求(这是本项目和"demo脚本"的本质区别)

用户有大量RJ音声要处理,不是跑一两个文件。以下要求所有模块必须满足,**这是本项目验收的核心标准,比功能本身更重要**:

1. **断点续跑**:任何模块,如果对应的输出文件已存在,默认跳过,不重新跑。要重新跑必须显式传 `--overwrite`。
2. **单文件失败不能拖垮整批**:处理几十上百个文件时,某一个文件失败(音频损坏、API超时等),记录到 `logs/failed.txt`,继续处理下一个,绝对不能让整个批次中断。
3. **失败可定位、可重跑**:`logs/failed.txt` 要写清楚失败的文件路径和具体报错,方便用户事后单独重跑这几个文件,不用重跑全部。
4. **进度可见**:批量跑几十个文件时要有进度提示(哪怕只是打印"处理中 12/87"),用户不应该对着黑屏不知道卡没卡死。
5. **路径要扛得住中文/特殊字符**:RJ文件夹和文件名经常包含日文、中文、特殊符号,所有路径处理必须考虑编码问题,这是已知高风险点。

## 6. 已有字幕文件的处理策略

部分RJ作品自带官方日文文本(常见为txt台本,少数为PDF),这是**质量优化的捷径,但不是必须依赖项**。处理原则:

### 6.1 两种已有文本的区别

| 类型 | 特征 | 用法 |
|---|---|---|
| txt纯文本台本 | 通常没有时间戳,只是逐句台词 | 不能直接转LRC(LRC核心是时间戳),但可用于**校对**——转录完日文后,对比官方文本纠正人名/专有名词/拟声词识别错误 |
| PDF场景标注 | 少数作品按场景分段标注大致时间点 | 时间精度通常不够细到逐句LRC需要的程度,仍需跑ASR补全精确时间轴,PDF文本同样用于校对 |

**结论:已有文本第一版定位是"校对参考",不是"替代ASR"。** 用文本去对齐生成时间戳(forced alignment)工程量较大,不在第一版范围内,留到后续质量优化阶段再排期。

### 6.2 架构上的要求:批处理总控必须做"已有字幕检测"

`modules/batch.py` 扫描每个RJ文件夹时,必须检测同目录下是否存在 `.txt` / `.pdf` 等文本文件,并记录检测结果(哪些作品有官方文本可用),具体处理逻辑:

```
1. 检测到已有文本 → 记录到 cache/has_reference_text.json 或类似清单,标记该作品"有参考文本"
2. 第一版批处理流程不强制使用该文本,ASR和LRC生成流程完全独立运行,不依赖它是否存在
3. 不允许假设"有文本就不用转录"——必须保证纯音频、无任何参考文本的作品也能独立走完整链路
```

**这条规则的原因**:不是所有RJ都有文本,管线的鲁棒性不能建立在"文本存在"这个不稳定前提上。已有文本是锦上添花的加分项,后续做"用文本校对ASR结果"或"用文本辅助强制对齐"这类优化时再接入,不要在第一版就让批处理逻辑依赖它。
